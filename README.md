# Notification Service — кейс на рефакторинг

Планировщик отложенных уведомлений. HTTP API, in-memory очередь, 100k+ задач,
много одинаковых timestamp и частый polling готовых к отправке уведомлений.

Сервис работает и проходит базовые тесты, но в проде ведёт себя плохо: worker
таймаутит на `GET /v1/due`, отменённые задачи продолжают отравлять hot path,
повторные `id` создают дубликаты, а порядок выдачи нестабилен. Ваша задача —
разобраться почему и привести сервис в чувство, не меняя публичное API.

## Бизнес-контекст

Продукт отправляет пользователям email, push и sms: напоминания об оплате,
обновления доставки, промо и security alert. Внешний worker периодически забирает
из сервиса пачку уведомлений, которые пора отправить, отправляет их и помечает
как `sent`.

API менять нельзя:

| Method | Path | Описание |
|---|---|---|
| `POST` | `/v1/notifications` | Запланировать уведомление |
| `GET` | `/v1/notifications/{id}` | Получить уведомление |
| `DELETE` | `/v1/notifications/{id}` | Отменить уведомление |
| `POST` | `/v1/notifications/{id}/sent` | Пометить уведомление отправленным |
| `GET` | `/v1/due` | Готовые к отправке: `?now=...&limit=...` |
| `GET` | `/health` | Healthcheck |

```bash
# 1. Сгенерировать датасет

./build/notification_service

```

## Что нужно сделать

Сервис «работает», но в проде регулярно прилетают жалобы. Ниже — реальные истории
из инцидентов. Внутреннюю структуру данных можно менять как угодно, публичное API —
нельзя.

### 1. SRE: «Worker начал таймаутить на due»

`GET /v1/due?now=...&limit=100` на большом наборе уведомлений проходит по всей
истории и сортирует всех кандидатов, хотя worker просит только первые 100 задач.

Нужно добиться, чтобы маленький `limit` реально ограничивал объём hot-path работы.
Подумайте про `std::set`, heap, iterator-index и стоимость удаления произвольной
задачи.

### 2. Backend: «Высокоприоритетные уведомления приходят позже обычных»

Ожидаемый порядок выдачи `due`:

1. `send_at asc`
2. `priority desc`
3. `created_at asc`
4. `id asc`

Baseline сортирует неполно. На одинаковом `send_at` порядок становится неправильным
и зависит от деталей реализации.

### 3. QA: «Повторный POST с тем же id создаёт дубль»

Клиент ретраит `POST /v1/notifications` после сетевого таймаута. Сервис должен быть
идемпотентным по `id`: повторный запрос не должен создавать вторую запись в due-очереди
и не должен перетирать уже запланированное уведомление неявно.

Сейчас в hot-path структуре остаются дубликаты `id`, а `GET /v1/due` может вернуть
одно уведомление несколько раз.

### 4. SRE: «Отменённые уведомления всё ещё тормозят систему»

`DELETE` только добавляет id в отдельный набор отменённых задач. После массовых отмен
`GET /v1/due` продолжает обходить старые записи и проверять их на каждом запросе.

Нужно решить, где физически удалять задачу из индекса готовности, и как не превратить
lazy deletion в бесконечный мусор.

### 5. Product: «Отправленное уведомление нельзя запланировать заново»

Для некоторых сценариев бизнес создаёт новое уведомление с тем же внешним `id` после
терминального состояния старого. После `sent` или `cancelled` новый `POST` должен
снова создать активную `pending` задачу, а старые terminal markers не должны её
скрывать.

При этом повторный `POST` для уже активной задачи должен оставаться идемпотентным.

### 6. SRE: «P99 растёт, хотя CPU простаивает»

`GET /v1/due`, `POST /v1/notifications`, `DELETE` и `POST /sent` сериализуются на
одном mutex. Под нагрузкой worker polling блокирует ingestion и наоборот.

Подумайте, где нужны exclusive locks, где достаточно read locks, и как держать
critical section короткой, не ломая согласованность индексов.

### 7. Backend: «No-ready запрос дорогой»

Когда все уведомления запланированы на будущее, `GET /v1/due` всё равно сканирует
всю коллекцию. Для очереди по времени такой запрос должен быстро понять, что готовых
задач нет.

## Бонусные задания

Опционально, для тех, кому мало:

- **Claim API**:
  `POST /v1/due/claim?now=...&limit=...` переводит `pending -> processing`, чтобы
  несколько worker-ов не отправляли одну задачу одновременно.
- **Retry policy**:
  `POST /v1/notifications/{id}/failed` увеличивает `attempts` и планирует retry с
  exponential backoff.
- **Rate limit by channel**:
  не больше N email/push/sms за окно времени.
- **Quiet hours**:
  перенос уведомлений пользователя, если `send_at` попал в ночной интервал.
- **Snapshot/load**:
  периодическое сохранение очереди и восстановление после рестарта.

## Бенчмарки baseline

Baseline намеренно наивный. На больших данных должны быть видны:

- полный проход по `schedule_`;
- полная сортировка candidates перед `limit`;
- общий mutex на чтение и запись;
- накопление cancelled/sent markers;
- дубликаты `id` в hot-path структуре.

Микробенчмарки:

| Бенчмарк | Что показывает |
|---|---|
| `BM_Due_SmallLimit` | выдача первых 100 due-задач |
| `BM_Due_NoReady` | запрос, когда готовых задач нет |
| `BM_Add_Throughput` | стоимость планирования новых уведомлений |
| `BM_MarkSent` | стоимость terminal transition |
| `BM_CancelledDoNotPoisonDue` | влияние массовых отмен на due |

Нагрузочный сценарий (`load_test.py`, `concurrency=50`) смешивает `GET /due`,
`POST /notifications`, `POST /sent` и `DELETE`. На baseline p99 должен быть заметно
хуже, чем после перехода к индексированной очереди.

## Критерии оценки

- **Корректность.** Все фиксированные тесты (`./build/notification_tests`) проходят,
  включая ранее `DISABLED_*` — после ваших правок они должны включиться и тоже
  зеленеть.
- **Порядок.** `due` строго сортируется по `send_at asc`, `priority desc`,
  `created_at asc`, `id asc`.
- **Идемпотентность.** Повторный `POST` активной задачи не создаёт дубль и не меняет
  существующую задачу случайно.
- **Terminal states.** `sent` и `cancelled` задачи не возвращаются из `due`, но новый
  `POST` после terminal state может снова создать активную задачу.
- **Производительность.** На том же датасете и сценарии нагрузки:
  - Total RPS вырос минимум в 5x.
  - p99 `GET /due` снизился минимум в 10x.
  - `BM_Due_SmallLimit` и `BM_Due_NoReady` показывают улучшение хотя бы на порядок.
- **Чистота.** Понятные индексы по `id` и времени готовности, без дублирующейся
  логики статусов.
- **Безопасность.** TSan-джоба остаётся зелёной, многопоточные тесты не падают.
- **Память.** Массовые отмены и отправки не оставляют бесконечный мусор в hot path.
